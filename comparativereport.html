Exercise: Research and Classification

This page documents 8 tools from different categories, classified by runtime requirements and annotated with personal notes and a critical reflection based on the ATELIER prompts.

Tool Grid

Legend: Node.js No Node.js Ruby Standalone SaaS

  • Tailwind CSS

    Node.js

    Category: Stylesheets / Utility-first CSS

    A utility-first CSS framework that encourages composing utility classes to build designs rapidly.

    Personal notes

    Fast for prototyping when using PostCSS and build step. Requires Node.js for the typical toolchain (jit, purge). Great for responsive work; slightly verbose HTML but highly productive.

  • Bootstrap

    No Node.js

    Category: Stylesheets / CSS Framework

    Popular component-based CSS framework with ready UI patterns and grid system.

    Personal notes

    Good for quick mockups (via CDN). Less flexible than utility-first approaches but simpler dependency management — no local build necessary.

  • Jekyll

    Ruby

    Category: Static Site Generator

    A Ruby-based static site generator (used by GitHub Pages historically).

    Personal notes

    Easy to deploy via GitHub Pages (legacy). I avoid adding Ruby to my machine unless necessary — it adds complexity but is stable and simple for Markdown-driven content.

  • Hugo

    Standalone

    Category: Static Site Generator

    A fast static site generator distributed as a single binary (Go-based).

    Personal notes

    Super fast local builds and minimal dependencies. Nice for large content sites; templates are Go-based which has a learning curve but is dependable for performance.

  • Material UI (MUI)

    Node.js

    Category: Component library / UI kit (React)

    A React UI library implementing Material Design components; used in modern web apps.

    Personal notes

    Excellent for React-based portfolios or dashboards. Adds Node dependency and bundler, but provides polished, accessible components.

  • shadcn/ui

    Node.js

    Category: Component Library (Radix + Tailwind)

    A collection of accessible components built on Radix, styled with Tailwind; intended for React + Next.js.

    Personal notes

    Great for consistent design systems; assumes Node.js and Tailwind. High flexibility but heavier setup than CDN-based approaches.

  • Eleventy (11ty)

    Node.js

    Category: Static Site Generator

    A simpler Node.js-based static site generator focusing on simplicity and flexibility.

    Personal notes

    My favourite when I want templating flexibility without a heavy JS framework. Requires Node but you can keep the setup light.

  • GSAP

    No Node.js

    Category: Animation / Visual Libraries

    GreenSock Animation Platform — powerful JS animation library used in production sites.

    Personal notes

    Works via CDN and integrates with vanilla JS. Ideal for portfolio interactions where a build step is unwanted.

  • Framer Motion

    Node.js

    Category: Animation / React

    Animation library designed for React apps — smooth declarative animations for components.

    Personal notes

    Perfect for React-based portfolios. Node.js required, but yields highly polished micro-interactions.

  • ImageKit.io

    SaaS

    Category: Data & Media / CDN

    A hosted image CDN & optimization service that resizes and serves images globally.

    Personal notes

    SaaS — no local install. Great to speed up image delivery in portfolios and reduce build-time image processing.

  • Hygraph

    SaaS

    Category: Headless CMS / Data

    Cloud-based GraphQL CMS for structured content and media.

    Personal notes

    Useful for content-driven portfolios that need a CMS without maintaining backend infra. Adds external dependency but simplifies content updates.

Personal Notes — Quick Summary

  • Prefer Node.js tools when building a component-driven React portfolio (MUI, Framer, shadcn) because they integrate with modern workflows.
  • Prefer No-Node approaches for quick prototypes or when sharing simple demos (Bootstrap CDN, GSAP CDN).
  • Standalone (Hugo) is my go-to when I want speed and minimal toolchain friction.
  • SaaS (ImageKit, Hygraph) remove infra work but create external runtime/service dependencies.

Critical Reflection (ATELIER)

A — Which tools fit my workflow?

I like a hybrid approach: Eleventy or Hugo for static content (fast builds), Tailwind + shadcn for consistent UI, and ImageKit for image delivery. For interactive micro-interactions I use GSAP when avoiding React, otherwise Framer Motion in React projects.

T — Trade-offs observed

  • Speed vs. Flexibility: Hugo + CDN CSS is fastest to iterate; Node.js + React gives more flexibility but slower initial setup and larger dependency surface.
  • Dependencies vs. Simplicity: Node-based stacks offer modern features but require managing package.json, lockfiles, and local Node versions; standalone binaries or CDN-first approaches reduce that complexity.
  • Node.js vs. Standalone: Node.js enables modern bundlers, hot reload, and component ecosystems. Standalone (Hugo) trades some ecosystem features for build speed and simplicity.
  • Creativity vs. Convention: Libraries like MUI or Bootstrap provide conventions (fast to deliver) but can limit visual uniqueness unless heavily customized. Tailwind/shadcn encourages unique visual language at the cost of initial setup.

E — How would I use them in my portfolio?

Primary portfolio (public): Hugo for fast builds + Tailwind for consistent styles + ImageKit for optimized images. For case-studies that need interactivity, create subprojects using React + shadcn + Framer Motion and deploy as separate apps or subpaths.

L — Limitations & Lessons

Tooling decisions are driven by maintenance and deployment targets: if I expect frequent content edits, use a SaaS headless CMS (Hygraph) or a lightweight CMS. If I need pixel-perfect components and animations, invest time in React-based setups.

I — Implementation plan (mini)

  1. Create a Hugo starter with Tailwind integrated (no Node for Hugo but Tailwind typically needs Node — weigh options: use precompiled CSS or add Node).
  2. Use ImageKit for responsive images and Hygraph for content management if clients need editing capability.
  3. For interactive case-studies, create a Next.js sub-app (Node required) with shadcn + Framer Motion.

R — Reflection summary

Pragmatically, I will mix approaches: keep the main portfolio simple and fast (Hugo or Eleventy) while using Node.js-based subprojects when greater interactivity and component reuse are needed.
